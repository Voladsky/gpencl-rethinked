# GPENCL - GPU-Accelerated Encryption Library

A Vulkan-based GPU-accelerated encryption library with ChaCha20-Poly1305 AEAD support.

## Overview

GPENCL provides GPU-accelerated file encryption and decryption using the ChaCha20-Poly1305 authenticated encryption cipher. The library uses Vulkan compute shaders to perform encryption operations on the GPU, with PBKDF2-based key derivation from passwords.

## Features

- **GPU-Accelerated Encryption**: ChaCha20-Poly1305 AEAD cipher implemented as Vulkan compute shaders
- **Password-Based Key Derivation**: PBKDF2-like key derivation with 10,000 iterations
- **File and Buffer APIs**: Support for encrypting both complete files and in-memory buffers
- **Multi-File Support**: Encrypt/decrypt multiple files with different passwords in batch operations
- **Device Probing**: Query and list available Vulkan-capable devices
- **Cross-Platform**: Windows (MSVC) and Unix (Clang/GCC) support via CMake

## Building

- Vulkan SDK 1.0+

## Encrypted File Format Specification

This document describes the GPENCL encrypted file format, following RFC-style documentation.

### Format Overview

GPENCL encrypted files follow a simple binary format:

```
┌──────────────┬──────────┬──────────┬────────┐
│   Metadata   │  Nonce   │   Data   │  Tag   │
├──────────────┼──────────┼──────────┼────────┤
│    8 bytes   │ 12 bytes │ Variable │16 bytes│
└──────────────┴──────────┴──────────┴────────┘
```

### Format Specification

#### Section 1: Original Length Header (8 bytes)

**Bytes 0-7: Original Plaintext Length**

```
Type:       uint64_t (unsigned 64-bit integer)
Byte Order: Little-endian
Range:      0 to 18,446,744,073,709,551,615 bytes

Description:
  Stores the original unencrypted file length before GPU processing.
  Used by the decoder to trim padded output to exact original size.
  Enables reconstruction of files with precise byte counts.
```

#### Section 2: Nonce (12 bytes)

**Bytes 8-19: ChaCha20 Nonce**

```
Type:       12 random bytes
Generation: Cryptographically random (unique per encryption)

Description:
  Nonce (Number Used Once) ensures the same plaintext encrypted
  multiple times produces different ciphertexts. Critical for security:
  reusing nonce+key with different plaintexts breaks encryption.
  Randomly generated on each encryption operation.
```

#### Section 3: Ciphertext (variable length)

**Bytes 20 to (20 + encrypted_length): Encrypted Data**

```
Type:       Binary ciphertext (encrypted plaintext)
Mechanism:  ChaCha20 stream cipher XOR
Alignment:  Padded to 256-byte boundary for GPU processing

Description:
  Plaintext encrypted using ChaCha20 keystream.
  Key derived from password using PBKDF2 (10,000 iterations).
  Poly1305 authentication computed over ciphertext length + data.
  Padding bytes are generated by the GPU and trimmed during decryption
  using the stored original length.
```

#### Section 4: Authentication Tag (16 bytes)

**Final 16 bytes: Poly1305 MAC Tag**

```
Type:       Poly1305 HMAC output (16 bytes)
Computed:   MAC(key, original_length || nonce || ciphertext)

Description:
  Authenticates the entire encrypted message.
  Verifies that encrypted data has not been modified.
  Prevents tampering and ensures message integrity.
  Tag verification must succeed for decryption to proceed.
```

### Complete Binary Layout

```
Offset  Size  Field              Description
──────  ────  ─────              ───────────
0       8     Original Length    uint64_t, little-endian
8       12    Nonce              Random bytes for ChaCha20
20      N     Ciphertext         Encrypted plaintext (padded to 256B boundary)
20+N    16    Poly1305 Tag       Authentication tag
```

### Example: 13 Bytes Encrypted

Encrypting plaintext "Hello, World!" (13 bytes):

```
Byte Offset  Size  Content
───────────  ────  ──────────────────────────────────
0-7          8 B   0x0d 0x00 0x00 0x00 0x00 0x00 0x00 0x00
             ↑ 13 (0x0d) in little-endian

8-19         12 B  [12 random bytes for nonce]
             Example: 42 3a 7f c1 d8 e0 55 92 a1 c8 39 5d

20-275       256 B [ChaCha20 encrypted data, padded]
             256 bytes because GPU aligns to 256-byte boundaries

276-291      16 B  [Poly1305 authentication tag]
             Example: c7 a2 f9 14 2e 5d 8c 90 a4 19 7b 3e
             ↑ Always 16 bytes, computed over all prior data

Total File: 292 bytes
```

### Security Properties

**Encryption Strength**
- ChaCha20 stream cipher: 256-bit security
- Poly1305 authentication: 128-bit security
- Overall security: 128-bit (limited by auth tag)

**Key Derivation**
- Algorithm: PBKDF2-like construction
- Iterations: 10,000
- Output: 256-bit (32 bytes)
- Salt: Fixed 8-byte constant

**Nonce Security**
- Generated: Cryptographically random
- Reuse: NEVER reuse same nonce with same key
- Impact: Nonce reuse breaks ChaCha20 security completely

### Compatibility

- **Format Version**: 1.0
- **Byte Ordering**: Little-endian (x86-64 native)
- **Encoding**: Raw binary (no text conversion)
- **File Signature**: None (raw format, no magic bytes)
- **Compression**: Uncompressed

## Command-Line Usage

### Probe Available Devices

```bash
./gpencl probe
```

Lists all Vulkan-capable GPU devices.

### Encrypt Files

```bash
./gpencl encrypt -p password1 file1.txt
./gpencl encrypt -p pwd1 file1.txt -p pwd2 file2.txt
./gpencl encrypt -pf passwords.txt file1.txt file2.txt
```

- `-p <password>` adds a password for a specific file.
- `-pf <password_file>` loads passwords from a file, one password per line.

Output files created with `.enc` extension.

### Decrypt Files

```bash
./gpencl decrypt -p password1 file1.txt.enc
./gpencl decrypt -p pwd1 file1.txt.enc -p pwd2 file2.txt.enc
```

Output files have `.enc` extension removed, or `.dec` if ambiguous.
